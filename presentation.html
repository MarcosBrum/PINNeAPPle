Software Fragmentation in PIML

Current tools force researchers to re-engineer the same infrastructure for every problem.

Pain Points

  • Re-implement loss functions with autodiff from scratch for every new problem
  • Manually manage collocation sampling strategies
  • Balance competing loss components — one of the biggest hyperparameter bottlenecks
  • Re-engineer inference & deployment infrastructure for each application
  • No single tool covers the full lifecycle: from problem specification to a live digital twin

Design
04 / 23

4 Design Principles

① Problem-First Design

ProblemSpec is the lingua franca — connects every module to the same symbolic problem definition. One object, consumed identically everywhere — including numerical solvers and visualizations.

② Full-Lifecycle Coverage

From geometry and voxelization, through agnostic numerical solving, GPU-accelerated PINN training, CFD-style visualization, UQ, digital twins, and REST serving — no gaps in the workflow.

③ Architectural Breadth

76 model implementations across 13 families, plus FDM/FEM/FVM/RBF numerical solvers — all registered and swappable — enabling fair comparison on identical problems.

④ Production Readiness

AMP mixed-precision, DistributedDataParallel, TorchScript/ONNX export, Zarr streaming data loaders, live digital twin runtime, and CFD-quality visualization — built in from day one.

Core API
06 / 23

ProblemSpec: The Central Abstraction

Captures everything needed to define a well-posed PDE problem in a single serializable object.

from pinneaple_environment import (
    get_preset, ProblemBuilder, DirichletBC
)

# Pathway 1: single-line preset
spec = get_preset("burgers_1d", nu=0.01)

# Pathway 2: fluent builder
spec = (
    ProblemBuilder("poisson_2d")
    .pde("poisson", fields=("u",), coords=("x", "y"))
    .param("f", 1.0)
    .domain_bounds({"x": (0,1), "y": (0,1)})
    .boundary_condition("wall", DirichletBC({"u": 0.0}))
    .build()
)

Consumed Identically By

  • Collocation sampler & voxel grid builder
  • FDM / FEM / FVM / RBF numerical solvers
  • Physics loss compiler
  • Validation & Arena benchmark runner
  • CFD-style visualization pipeline

Key Benefit

Eliminates transcription errors when re-specifying physics across tools. Define once, use everywhere — including the numerical reference solver.

Voxelization & Meshfree Sampling New

Convert any geometry representation into collocation points without meshing.

pinneaple_geom.ops.voxelize

from pinneaple_geom.ops import (
    voxelize_domain, voxelize_sdf,
    voxelize_pointcloud, sample_voxelgrid
)

# From domain bounds
grid = voxelize_domain(
    {"x": (0,1), "y": (0,1), "z": (0,1)},
    resolution=64
)

# From SDF — geometry-aware
grid = voxelize_sdf(my_sdf_fn, bounds, 128)

# Collocation points for training
pts = grid.to_collocation_points()
# or subsample: "random" | "all" | "surface"
pts = sample_voxelgrid(grid, 10000, strategy="surface")

VoxelGrid

Stores data tensor + origin + voxel_size. Methods: to_world(ijk), to_voxel(xyz), occupied_centres().

Three Voxelization Sources

SourceFunction

Signed-Distance Fieldvoxelize_sdf(sdf_fn, bounds, res)
Point cloudvoxelize_pointcloud(pts, bounds, res)

Every solver reads ProblemSpec directly — no manual assembly or problem-specific code.

# One API for every solver
from pinneaple_solvers.fdm import FDMSolver

solver = FDMSolver.from_problem_spec(spec, nx=128, ny=128)
out = solver.solve_from_spec(spec)  # → SolverOutput

# Auto-routed in problem_runner
dataset = generate_pinn_dataset(spec, n_data=5000)

Dispatch on spec.pde.kind

Zero boilerplate

Each solver reads pde.kind, pde.params, domain_bounds, and conditions automatically. Boundary conditions from DirichletBC/NeumannBC are applied without user intervention.

CFL-stable time stepping

FDM and FVM compute dt automatically from the grid and PDE parameters. FEM uses direct or CG linear solve with Tikhonov regularisation.

SolverOutput

result tensor · losses dict (residual) · extras (nodes, elems, xc, yc, dt, …)

Meta-Learning

Meta-Learning for PDE Families

Goal: rapid adaptation to an unseen member of a PDE family (e.g., Burgers at a new viscosity ν).

MAML (Model-Agnostic Meta-Learning)

θ'i = θ − α·∇θℒτᵢ(θ)

outer: minθ Σi ℒτᵢ(θ'i)

Reptile

First-order variant — no second-order derivatives. Substantially lower memory than full MAML.

Adapt a New Config in Seconds

# After meta-training on a family:
meta_adapt(task, n_steps=20)

PDETaskSampler

Generates tasks by sampling physical parameters from param_ranges and compiling the loss via compile_problem — no manual task construction needed.

Conclusion
23 / 23

Conclusion

8 Key Differentiators

  • Unified ProblemSpec — one object consumed identically by every module
  • Problem-agnostic FDM / FEM / FVM — solvers auto-read ProblemSpec, no boilerplate
  • Meshfree methods — RBF collocation (Kansa) and Moving Least Squares for scattered nodes
  • Voxelization pipeline — SDF / point cloud / domain → collocation points in one call
  • CFD-style visualization — scalar fields, streamlines, FEM/FVM plots, GIF animations
  • Zarr-backed streaming — training on out-of-memory datasets
  • Live digital twin runtime — multi-protocol sensor fusion, Kalman assimilation
  • Distribution-free conformal UQ — coverage guarantees without distributional assumptions

Roadmap

  • JAX and MLX backend support
  • Automated multi-backend Arena with public leaderboard
  • HPC/cloud integration (SLURM, Ray)
  • Interactive 3-D visualization via Plotly/VTK
